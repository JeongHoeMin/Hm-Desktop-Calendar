# 240 클라이언트 계정 관리 UI

## 상태

- 상태: 완료
- 브랜치: `feat/account-management-client`

## 목표

로그인한 사용자가 앱 안에서 비밀번호 변경과 계정 삭제를 수행할 수 있게
한다.

## 의존성

- 230 (계정 관리 API가 `main`에 병합되고 서버에 배포 가능해야 한다)

## 범위

- `Authentication/AuthSession.cs`에 `ChangePasswordAsync`와
  `DeleteAccountAsync`를 추가한다.
- ☰ 메뉴의 로그인 상태에 "계정" 항목을 추가하고 계정 창을 연다.
  - 비밀번호 변경 폼: 현재/신규/확인 입력.
  - 계정 삭제: 확인 문구 입력 후에만 버튼 활성화.
- 변경 성공 시 세션을 정리하고 재로그인을 안내한다.
- 삭제 성공 시 로그아웃하고 로컬 계정 스코프 데이터 삭제 여부를 선택받는다.

## 제외 범위

- 서버 변경(230에서 완료).
- 이메일 변경.

## 설계 결정

- 계정 창은 `LoginWindow`와 같은 단일 인스턴스 패턴을 따른다.
- 서버 오류는 한국어 메시지로 매핑한다(예: 401 → "현재 비밀번호가 일치하지
  않습니다").
- 삭제 확인 문구는 계정 이메일 입력으로 한다.
- 비밀번호 변경과 계정 삭제 성공 시 Windows 자격 증명의 갱신 토큰, 액세스 토큰과
  사용자 상태를 함께 지운다. 계정 삭제 뒤에는 별도 확인 대화상자에서 이 PC의
  계정 범위를 유지하거나 물리 삭제하도록 선택한다.
- 계정 창은 기본 680px 높이에서 스크롤되며 520×620 최소 크기에서도 모든 입력과
  작업 버튼에 접근할 수 있다.

## 완료 조건

- 비밀번호 변경 → 재로그인 → 동기화가 정상 동작한다.
- 계정 삭제 후 다른 기기의 세션도 다음 토큰 갱신 시점에 로그아웃된다.

## 검증

- 뷰모델 단위 테스트(입력 검증, 확인 문구 활성화 규칙, 오류 매핑).
- 로컬 서버를 대상으로 변경·삭제 수동 시나리오를 수행하고 PR에 기록한다.
- UI 변경이므로 PR에 스크린샷을 포함한다.

### 현재 검증 결과

- `dotnet build HmDesktopCalendar.csproj -c Debug`: 경고·오류 없이 성공.
- 전체 회귀 테스트 74개 통과. 입력 검증, 이메일 확인 활성화, 401 한국어 매핑,
  성공 후 세션 종료 안내와 선택한 로컬 계정 범위만 삭제하는 동작을 포함한다.
- 로컬 Docker 서버에서 실제 `AuthSession`으로 비밀번호 변경 후 로그아웃, 기존
  비밀번호 거부, 새 비밀번호 로그인, 계정 삭제 후 로그아웃과 삭제 계정 로그인
  거부를 확인했다. 테스트 토큰은 메모리 저장소를 사용해 Windows 자격 증명을
  변경하지 않았다.
- Windows 계정 창에서 초기 버튼 비활성화와 유효한 현재·신규·확인 비밀번호 및
  계정 이메일 입력 후 두 버튼 활성화를 확인했다. 정책상 UI의 최종 비밀번호 변경
  제출과 삭제 클릭은 수행하지 않고 동일 상태 전이를 실제 클라이언트 API 검증으로
  확인했다.
