# 000 작업 관리 구조와 기능 로드맵

## 상태

- 상태: 완료
- 브랜치: `docs/feature-roadmap`

## 목표

후속 기능을 한 번에 구현하지 않고, 의존성과 검증 범위가 명확한 작은 PR로 순차
진행할 수 있는 한국어 작업 관리 구조를 만든다.

## 의존성

- PR #2의 하네스 엔지니어링 정책과 제품 개요가 `main`에 병합돼 있어야 한다.

## 범위

- `inactive`, `active`, `completed` 상태 디렉터리와 사용 규칙을 작성한다.
- 재사용 가능한 한국어 작업 문서 템플릿을 작성한다.
- 합의된 기능을 의존성 기준으로 나눈 비활성 작업 문서를 등록한다.
- 각 작업에 목표, 범위, 제외 범위, 설계 결정, 완료 조건과 검증 방법을 기록한다.

## 제외 범위

- 애플리케이션 코드, 서버 코드, 데이터 모델과 디자인을 변경하지 않는다.
- 등록한 기능 작업을 구현하지 않는다.

## 설계 결정

- 상태 디렉터리 이름은 정책에 맞춰 `inactive`, `active`, `completed`를 사용한다.
- 작업 문서는 모두 한국어로 작성하고 세 자리 ID로 순서를 고정한다.
- 한 번에 활성 작업과 열린 구현 PR을 하나만 유지한다.
- 완료는 구현과 검증을 마치고 PR을 연 상태를 의미한다.

## 완료 조건

- 세 상태 디렉터리, 안내 문서와 템플릿이 존재한다.
- 현재 문서 하나만 활성 상태다.
- 후속 기능이 중복 없이 비활성 문서로 나뉘고 의존성이 명시돼 있다.
- 모든 새 Markdown 문서가 한국어다.

## 검증

- 문서 경로와 활성 문서 개수를 검사한다.
- 필수 항목 누락과 기능 요구 중복 여부를 검토한다.
- `git diff --check`와 스테이징 파일 목록을 확인한다.
