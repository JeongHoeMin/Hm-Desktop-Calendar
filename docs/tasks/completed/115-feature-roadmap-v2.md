# 115 기능 로드맵 v2 등록

## 상태

- 상태: 완료
- 브랜치: `docs/feature-roadmap-v2`

## 목표

완료된 010~110 이후의 후속 기능을 4개 기능군(달력 완성도, 설정·수명주기,
데이터 안전망, 서버·계정 고도화)으로 나누고, 의존성과 검증 범위가 명확한
작은 PR 단위의 비활성 작업 문서 세트로 등록해 순차 구현이 가능하게 한다.

## 의존성

- 110까지의 PR이 `main`에 병합돼 있어야 한다.
- 커밋된 `server/.env.example`에 실제 DB 접속 문자열이 노출된 사실을
  사용자에게 통지하고, DB 자격 증명 회전(운영 조치)을 권고한다. 저장소 쪽
  정리는 작업 120으로 등록한다.

## 범위

- `docs/tasks/inactive`에 120~250 작업 문서 14개를 `_template.md` 필수
  항목을 모두 채워 생성한다.
- 각 문서에 목표, 의존성(선행 ID), 범위, 제외 범위, 설계 결정, 완료 조건,
  검증을 한국어로 기록한다.
- 이메일 기반 기능(비밀번호 재설정·이메일 인증)과 ICS 가져오기의 제외
  사유를 해당 문서에 기록한다.
- 권장 구현 순서와 의존성 그래프를 이 문서에 기록한다.

## 제외 범위

- 애플리케이션·서버 코드, 데이터 모델, 디자인을 변경하지 않는다.
- 등록한 작업을 구현하지 않는다. 구현은 작업별로 별도 진행한다.
- `docs/tasks/README.md`, `docs/product-overview.md`와 000 문서를 수정하지
  않는다.

## 설계 결정

- 로드맵 등록 작업은 115, 기능 작업은 120부터 10 단위로 부여해 메타 작업과
  기능 작업을 구분한다.
- 서버와 클라이언트가 모두 필요한 기능(계정 관리)은 서버 선행(230)과
  클라이언트 후행(240) 2개 작업으로 나눈다.
- 여러 설정 작업의 공통 기반(설정 저장소 140, 설정 창·트레이 150)은 개별
  설정 작업(160, 170, 180, 250)과 분리한다.
- SMTP·메일러 인프라가 없는 현재 상태에서 이메일 발송이 필요한 기능은
  등록하지 않고, 인프라가 준비되는 시점에 별도 작업으로 등록한다.
