# Hermes 에이전트 개발 정책

## 적용 범위와 우선순위

이 문서는 저장소 전체에 적용되는 지속적인 개발 절차를 정의한다. 더 하위 경로의
`AGENTS.md`가 더 엄격한 지침을 제공하면 해당 하위 경로에서는 그 지침도 따른다.
현재 작업에 대한 사용자의 명시적 지시가 이 문서와 충돌하면 사용자 지시를
우선한다.

이 문서에는 특정 기능 요구사항, 임시 계획, 작업별 세부 내용을 기록하지 않는다.
그 내용은 `docs/tasks` 아래의 작업 문서에서 관리한다.

저장소에서 새로 작성하거나 수정하는 모든 Markdown 문서는 한국어로 작성한다.
코드 식별자, 명령어, 경로, 고유한 기술 용어는 원문 표기를 함께 사용할 수 있다.

프로그램의 목적, 현재 기능, 전체 시스템 구성이나 제품 경계를 확인해야 하는
작업에서는 `docs/product-overview.md`를 참고한다. 모든 작업에서 의무적으로 읽을
필요는 없다. 변경 사항이 해당 문서의 사실을 바꾸는 경우 같은 PR에서 함께
갱신한다. 작업 상태나 임시 구현 메모는 제품 개요에 기록하지 않는다.

## Hermes 작업 절차

모든 변경은 다음 순서를 따른다.

1. 저장소와 관련 외부 상태를 먼저 조사한다. 추측보다 소스 코드, 테스트, 스키마,
   설정과 공식 문서를 우선한다. 제품 맥락이 필요하면 제품 개요를 참고한다.
2. 작업 하나만 선택해 문서를 `docs/tasks/active`로 이동하거나 생성한다. 여러
   비활성 작업을 한 번에 구현하지 않는다.
3. 목표, 의존성, 범위, 제외 범위, 설계 결정, 완료 조건과 검증 방법을 확정한다.
4. 활성 작업을 만족하는 가장 작은 단위로 구현한다. 관련 없는 사용자 변경과
   범위 밖 코드는 건드리지 않는다.
5. 위험도에 맞게 테스트, 빌드, 마이그레이션 검사와 수동 시나리오를 실행한다.
6. 전체 diff를 검토해 정확성, 보안, 호환성, 비밀 정보, 불필요한 복잡도와 우발적
   파일 포함 여부를 확인한다.
7. 원자적인 Conventional Commit을 만들고 작업 브랜치를 푸시한 뒤 기본 브랜치를
   대상으로 PR 하나를 연다.
8. 구현과 검증이 끝나고 PR이 열렸을 때 작업 문서를 `docs/tasks/completed`로
   이동하고 커밋과 PR URL을 기록한다. 수정 요청을 받으면 다시 `active`로 옮긴다.
9. PR URL을 사용자에게 전달하고 멈춘다. 사용자를 대신해 승인하거나 병합하지
   않는다. 이전 PR의 병합을 사용자가 확인한 후에만 다음 작업을 시작한다.

## 작업 문서

작업 상태는 문서 위치로 표현한다.

- `docs/tasks/inactive`: 승인된 대기 작업
- `docs/tasks/active`: 현재 구현 또는 수정 중인 단 하나의 작업
- `docs/tasks/completed`: 구현과 검증을 완료하고 PR을 연 작업

각 작업 문서에는 ID, 상태, 목표, 의존성, 범위, 제외 범위, 설계 결정, 완료 조건,
검증, 커밋과 PR URL을 포함한다. 결정이 바뀌면 문서를 즉시 갱신한다.

작업 폴더가 아직 없다면 작업 관리 구조 생성 자체를 별도 문서 작업과 PR로
진행한다. 제품 구현과 섞지 않는다.

## Git과 GitHub

- 로컬 이력 작업에는 `git`, GitHub PR 작업에는 `gh`를 사용한다.
- 최신 기본 브랜치에서 전용 브랜치를 만든다. 브랜치 접두사는 `feat/`, `fix/`,
  `refactor/`, `test/`, `docs/`, `chore/` 중 하나를 사용한다.
- 기본 브랜치에 직접 커밋하지 않는다. 사용자의 명시적 허가 없이 공유 브랜치를
  강제 푸시하거나 공개 이력을 다시 작성하지 않는다.
- 활성 브랜치와 열린 구현 PR은 한 번에 하나만 유지한다.
- Conventional Commit 제목을 사용하고 각 커밋은 독립적으로 검토 가능해야 한다.
- Codex가 지원한 커밋에는 다음 Git 트레일러를 추가한다.

  `Assisted-by: OpenAI Codex`

- 저장소 소유자를 Git 작성자로 유지한다. Codex의 봇 계정이나 이메일을 임의로
  만들거나 사칭하지 않는다.
- PR 본문에는 요약, 작업 문서, 검증 결과, 마이그레이션 또는 호환성 영향, 시각적
  변경의 스크린샷, 후속 작업을 포함한다. OpenAI Codex의 지원 여부도 표기한다.
- 비밀 정보, 로컬 환경 파일, 의존성 폴더, 빌드·테스트 결과와 임시 파일을
  커밋하지 않는다.

## 구현 원칙

- 편집 전에 작업 트리를 확인한다. 활성 작업이 명시적으로 소유하지 않는 기존
  변경은 사용자의 것으로 취급한다.
- 기존 추상화와 이름을 우선한다. 새 계층은 실제 경계를 격리하거나 확인된 중복을
  제거할 때만 추가한다.
- 공개 계약은 명시적으로 유지하고, 활성 작업에 조정된 호환성 계획이 없는 한
  하위 호환성을 지킨다.
- 영구 데이터와 네트워크 계약 변경은 마이그레이션 작업으로 취급한다. 순방향
  마이그레이션, 기존 데이터 처리, 실패 동작과 호환성 테스트를 제공한다.
- 비동기 작업은 동기화, 취소, 종료, 재시도와 실패 상태를 명시적으로 다룬다.
- 변경 가능성이 있는 표준, 라이브러리, 플랫폼 동작과 외부 디자인 시스템은 공식
  1차 자료를 사용하고 중요한 출처는 작업 문서에 기록한다.
- 범위를 조용히 넓히지 않는다. 유용하지만 완료 조건에 필요 없는 일은 새로운
  비활성 작업으로 등록한다.

## 검증과 자체 리뷰

- 구현 중에는 가장 좁은 관련 테스트를 실행하고, 커밋 전에는 영향받는 전체
  테스트와 빌드를 실행한다.
- 수정한 버그에는 회귀 테스트를, 새 도메인 동작에는 결정론적인 단위 테스트를
  추가한다.
- 데이터 변경은 해당되는 마이그레이션, 기존 데이터, 삭제 tombstone, 충돌과
  동기화 시나리오를 검증한다.
- UI 변경은 지원하는 최소 크기에서 동작과 화면을 검토한다. 레이아웃이나 스타일이
  달라지면 PR에 스크린샷을 포함한다.
- 실행하지 못한 테스트를 통과로 취급하지 않는다. 필수 검증이 막히면 정확한 원인을
  기록하고 완료 PR을 열지 않는다.
- 푸시 전에 `git diff`, `git diff --check`, 스테이징 파일과 기본 브랜치 대비 최종
  커밋 범위를 확인한다.

## 완료 정의

완료 조건을 만족하고 필수 검증이 통과했으며 문서와 마이그레이션이 최신이고,
우발적 변경이 없고, 커밋이 이 정책을 따르며, 재현 가능한 검증 내용을 담은 PR이
열렸을 때만 사용자 검토 준비가 끝난 것으로 본다.
